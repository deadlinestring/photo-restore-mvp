# Правила разработки

- Не усложнять MVP без отдельной задачи.
- Не добавлять оплату, регистрацию, личный кабинет, тарифы и админку без прямого запроса.
- Все внешние API в будущем вызывать только на server-side.
- `SUPABASE_SERVICE_ROLE_KEY` использовать только в server-side коде.
- Не хранить секреты в коде.
- `.env.local` не коммитить.
- Пользовательский флоу должен оставаться простым: главная -> загрузка -> ожидание -> результат.
- После изменений проверять TypeScript/build, если команды доступны.
- Если меняются env-переменные, обновлять `.env.example` и `README.md`.
- `FAL_KEY` использовать только в server-side коде.
- Не запускать массовые fal.ai проверки.
- Не запускать fal.ai без прямого запроса.
- Для публичного запуска до подключения оплаты использовать `RESTORE_FLOW_MODE=payment_required`.
- Не запускать fal.ai до подтвержденной оплаты в `payment_required`.
- Не ставить `payment_status=paid` по нажатию кнопки на frontend.
- Успешная оплата должна подтверждаться только server-side/webhook.
- Если используется бренд KARMA в оплате, пользовательский интерфейс должен заранее объяснять это спокойно и понятно.
- Не подключать Prisma, дополнительные AI API или платежные сервисы без отдельной задачи.
- Не делать Supabase buckets публичными без прямого запроса.
- Сохранять тексты интерфейса на русском языке, если задача не требует другого.
- Держать UI легким: без тяжелых анимаций и лишних зависимостей.
