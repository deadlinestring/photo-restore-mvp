# Правила разработки

- Не усложнять MVP без отдельной задачи.
- Не добавлять регистрацию, личный кабинет, тарифы, пакеты и публичную админку без прямого запроса.
- Все внешние API вызывать только на server-side.
- `SUPABASE_SERVICE_ROLE_KEY`, `FAL_KEY` и `YOOKASSA_SECRET_KEY` использовать только в server-side коде.
- Не хранить секреты в коде.
- Не логировать платежные секреты.
- `.env.local` не коммитить.
- Если меняются env-переменные, обновлять `.env.example` и `README.md`.
- Пользовательский flow должен оставаться простым: главная -> загрузка -> checkout/оплата -> ожидание -> результат.
- После изменений проверять TypeScript/build, если команды доступны.
- Не запускать массовые fal.ai проверки.
- Не запускать fal.ai без прямого запроса.
- В публичном flow `RESTORE_FLOW_MODE` должен быть `payment_required`.
- В `payment_required` fal.ai нельзя запускать до `payment_status=paid`.
- Любая оплата подтверждается только server-side.
- Frontend не может ставить `payment_status=paid`.
- Нельзя ставить `payment_status=paid` по нажатию кнопки на frontend.
- Успешная оплата должна подтверждаться только server-side/webhook или server-side проверкой платежа через API платежного провайдера.
- YooKassa webhook/check endpoint должны перепроверять платеж через API YooKassa.
- `YOOKASSA_WEBHOOK_SECRET` не считать единственной защитой, если схема подписи не подтверждена документированным заголовком.
- Если используется бренд KARMA в оплате, пользовательский интерфейс должен заранее объяснять это спокойно и понятно.
- Не подключать Prisma, дополнительные AI API или платежные сервисы без отдельной задачи.
- Не делать Supabase buckets публичными без прямого запроса.
- Сохранять тексты интерфейса на русском языке, если задача не требует другого.
- Держать UI легким: без тяжелых анимаций и лишних зависимостей.
